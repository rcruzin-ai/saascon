# SPEC — calorie-tracker (v1)

> Generated by `/spec`. PRD for a single-user, local-first calorie tracker built on the saascon template (Next.js 15 + SQLite + Tailwind v4). Open questions in §9 must be resolved before `/plan`.

## 1. Objective

- **Problem:** logging food intake against a calorie goal needs to take seconds, on a phone, with no account or internet. Existing apps require sign-up, push notifications, and food databases the user doesn't trust.
- **Users:** a single individual tracking their own intake on their own machine. No multi-user, no auth, no sync.
- **Success metric:** logging a previously-eaten food takes ≤ 2 taps from the home page. Daily totals + remaining-against-target are visible above the fold on a 375 px viewport without scrolling.

## 2. Scope

### In scope (v1)
- [ ] **Today view (`/`)** — list of foods logged today with timestamp, running totals (calories, protein g, carbs g, fat g), progress bar against `daily_calorie_target`.
- [ ] **Quick-add form** — name + calories (required) + optional protein/carbs/fat in g. Logs the food and adds an entry for "now" in one submit. The single `'use client'` surface in v1.
- [ ] **Foods library (`/foods`)** — list of all previously-logged foods with macros. Each row has a one-tap "Log again" button (server action; no client JS).
- [ ] **Settings (`/settings`)** — single field: `daily_calorie_target`. Server-component form, redirects to `?ok=1` / `?error=invalid`.
- [ ] **History view (`/history`)** — last 7 days, one row per day with total calories + macro totals + a CSS bar showing calories vs target. Single aggregate query + JS-side day fill (no N+1).
- [ ] **Delete entry** — per-row server-action button on the today view (no confirm modal in v1; instant delete).

### Out of scope (explicit non-goals for v1)
- Auth / multi-user
- Meal categories (breakfast/lunch/dinner) — flat list per day
- Photo uploads, barcode scan, external food DB
- Macro targets, weight tracking, any goal beyond a single calorie target
- Sync across devices, cloud backup
- Charts beyond CSS bars (no chart library)
- Editing past entries (delete + re-log)

### Future / deferred
- Editable entries (right now: delete + re-log)
- Macro targets and per-macro progress
- Cloud sync (would activate the Supabase mode that already exists in the template)
- Search/filter in the foods library once it grows past ~30 rows

## 3. User stories

- **US-1** — As the user, I want to log "oatmeal, 320 kcal" in two taps so I can stop forgetting to track.
- **US-2** — As the user, I want to see my running totals against my calorie target on the home page so I don't have to navigate.
- **US-3** — As the user, I want previously-logged foods to be one tap to re-log so the second time is faster than the first.
- **US-4** — As the user, I want to set my daily calorie target once and have it persist so the progress bar means something.
- **US-5** — As the user, I want to see the last 7 days at a glance with a bar per day so I can spot trends.
- **US-6** — As the user, I want to delete an entry I logged by mistake without an edit screen.

## 4. Functional requirements

- **FR-1** — Quick-add: name (1–80 chars, trimmed), calories (integer 0–10000), protein/carbs/fat g (decimal 0–1000, optional). Validation on the server action; inline error via `useActionState`.
- **FR-2** — Quick-add upserts the food by case-insensitive name (`on conflict (lower(name)) do update`) so the library doesn't accumulate duplicates. Macros on the existing food row are updated to the latest submission.
- **FR-3** — Each `entries` row snapshots the four nutrition fields at log time (see §6). Editing a food's macros later does NOT change historical totals.
- **FR-4** — "Today" boundary uses server-local timezone. Range is `[startOfTodayLocal, startOfTomorrowLocal)` converted to UTC for the SQL `where`. Single-user app — server TZ = user TZ.
- **FR-5** — Re-log button on the foods library inserts a new `entries` row with `logged_at = now` and snapshot fields copied from the food's current values. One server action, redirects back to `/foods`.
- **FR-6** — History view groups by `date(logged_at, 'localtime')` over `[today-6, today]`. Single aggregate query, JS fills missing days with zeros. Bar width = `min(100, total/target * 100)` percent; bar color shifts past 100%.
- **FR-7** — Delete entry removes the entries row by id, server action, redirects back to `/`.
- **FR-8** — Daily target settable via `/settings`. Stored in `settings` row with `id = 1` (singleton). Default seeded value: 2000 kcal. Integer 500–10000.

## 5. Non-functional requirements

- **Performance:** First Load JS ≤ 1.3 kB on the page hosting the quick-add form, ≤ 170 B on every other page (no rogue `'use client'`). History page issues exactly one aggregate SQL query — verified via `grep -c '.prepare(' …` returning 1.
- **Accessibility:** WCAG 2.1 AA. Tap targets ≥ 44 × 44 px. `<label>` on every input. `role="progressbar"` with `aria-valuenow/min/max` on the target bar. `role="alert"` on validation errors. Focus rings via `focus-visible:ring-*`.
- **Browser support:** modern evergreen + iOS Safari (mobile-first is the primary use case).
- **Privacy / threat model:** no auth, no telemetry, no network. All data lives in `template/local.db`; deleting that file deletes everything. v1 ships server actions for create / re-log / delete / settings — all of them are publicly invokable on `localhost:3000`; the only barrier is same-origin browser convention. This is acceptable for a single-user local app on a personal machine. **If `cloud` mode is ever activated, every write path needs an auth gate or anon-write RLS policy** — the current schema grants anon `read` only.
- **SQL safety:** zero string interpolation in `prepare(...)`. Verified via `grep -nE 'prepare\(.*\$\{|prepare\(.*\+'` returning zero hits.

## 6. Data model (high-level)

Three entities. Detailed schema in `db/sqlite/schema.sql` and `db/schema.sql`; ER diagram in `docs/er-diagram.md`.

- **`foods`** — the library. `(id text pk, name text not null, calories integer not null, protein_g real, carbs_g real, fat_g real, created_at text default current_timestamp)`. Unique index on `lower(name)` so the upsert pattern works.
- **`entries`** — log events. `(id text pk, food_id text references foods(id) on delete set null, name_snapshot text, calories_snapshot integer, protein_snapshot real, carbs_snapshot real, fat_snapshot real, logged_at text default current_timestamp)`. **Snapshot semantics** — historical rows do not change when the food's catalog row is later edited. `food_id` retained for "re-log this food" affordance; nullable so deleting a food doesn't destroy history.
- **`settings`** — singleton. `(id integer pk check (id = 1), daily_calorie_target integer not null default 2000)`. Seeded with one row in the schema migration.

## 8. External dependencies

- **None.** SQLite via `better-sqlite3` against `./local.db`. No Supabase, no Vercel, no chart library, no date library, no form library.
- Stack pinned per TEMPLATE.md §2: Next.js 15, React 19, TypeScript strict, Tailwind v4, npm.

## 9. Open questions — resolved

- [x] **Q1 — Snapshot vs live-link on entries.** **Decision: snapshot.** Each `entries` row stores `name_snapshot`, `calories_snapshot`, `protein_snapshot`, `carbs_snapshot`, `fat_snapshot`. Editing a food's catalog row does not change historical totals. Captured in FR-3 and §6.
- [x] **Q2 — Timezone for "today".** **Decision: server-local TZ.** Build `[startOfTodayLocal, startOfTomorrowLocal)` from `new Date()` and convert to UTC for the SQL where-clause. History uses `date(logged_at, 'localtime')`. Captured in FR-4 and FR-6.
- [x] **Q3 — Success metric.** **Decision: re-log in ≤ 2 taps + today-view totals above the fold on a 375 px viewport.** `/test` will assert (a) the foods-library re-log is one server-action button per row (one tap from `/foods`, two from `/`) and (b) the today-view JSX uses the mobile-first wrapper with no scroll-required layout above the entries list.
- [x] **Q4 — Default daily target on a fresh DB.** **Decision: 2000 kcal seeded default.** Schema migration inserts the `settings` row with `daily_calorie_target = 2000`. Progress bar renders immediately on first launch; user changes it on `/settings`.
